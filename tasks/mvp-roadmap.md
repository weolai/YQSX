# MVP 最小可运行商城闭环路线图

## 目标

基于 `p0-backlog.md` 重新规划任务顺序，优先形成最小可运行商城闭环（MVP）。

**MVP 核心流程：**

```
用户登录 → 查询商品 → 创建订单 → 完成支付
```

---

## 一、为什么调整顺序

### 1.1 原 P0-backlog 的问题

原 `p0-backlog.md` 按服务维度分组（用户、商品、订单、支付），每个服务内部串行开发。这导致：

- **闭环形成过晚**：必须等所有服务全部开发完成，才能跑通一次完整购物
- **反馈周期长**：无法在中期验证架构是否合理
- **风险堆积**：接口联调、跨服务问题在最后阶段才暴露
- **非核心功能占用早期时间**：注册、商品删除、订单取消、DTO/VO 规范化等不阻塞主流程

### 1.2 MVP 策略的优势

| 维度 | 原策略 | MVP 策略 |
|-----|--------|---------|
| 目标 | 按服务完善 | 按流程闭环 |
| 反馈周期 | 长（数周） | 短（一个阶段即可演示） |
| 风险 | 后期集中爆发 | 早期暴露 |
| 可演示性 | 差 | 每阶段可演示 |
| 信心建立 | 慢 | 快 |

### 1.3 调整原则

1. **深度优先**：优先打通一个用户场景，而非广度覆盖所有功能
2. **最小可用**：第一阶段结束后必须能完成 `登录 → 查商品 → 下单 → 支付`
3. **依赖先行**：严格尊重 `p0-backlog.md` 中的依赖关系，避免阶段内出现阻塞
4. **非阻塞后置**：不阻塞主流程的功能延后

## 二、MVP 阶段规划

### 阶段总览

| 阶段 | 目标 | 核心任务 | 预计工时 | 里程碑 |
|-----|------|---------|---------|--------|
| 阶段一 | 搭建最小可运行闭环 | 数据库 + 登录/商品/订单/支付基础模块 + 余额扣减 | 40小时 | 能登录、能查商品、能下单、能支付 |
| 阶段二 | 完善核心交易流程 | 注册、库存扣减、订单取消、订单明细、全局异常 | 21.5小时 | 主流程稳定可用 |

## 三、阶段一：最小可运行闭环（40小时）

### 目标

实现 `登录 → 查商品 → 创建订单 → 完成支付` 的最小闭环。

### 关键依赖处理

- `P0-002-11`（余额扣减）依赖 `P0-002-10`（用户信息更新），`P0-002-10` 依赖 `P0-002-09`（用户信息查询）。为在阶段一形成支付闭环，将 `P0-002-09`、`P0-002-10`、`P0-002-11` 全部提前到阶段一。
- `P0-004-03`（订单支付接口）依赖 `P0-005-06`（支付 Controller），是支付闭环的最后一步，必须放在阶段一。

- 库存扣减 `P0-003-05` 不阻塞"创建订单"本身，延后到阶段二。

### 阶段一验收标准

**必须完成端到端流程验证：**

1. 用户登录：调用 `/user/login` 返回用户信息和 Token（阶段一暂不强制 Sa-Token，可用简单 Token 或 session）
2. 查询商品：调用 `/product/list` 返回商品列表
3. 创建订单：调用现有 `/orders/save` 创建订单
4. 完成支付：调用 `/order/pay` 触发支付，内部调用支付服务扣减余额
5. 支付后校验：调用 `/order/{id}` 查询订单状态为已支付，用户余额减少

**可演示场景：**
```
POST /user/login        -> { userId, token }
GET  /product/list      -> [ { productId, name, price, stock } ]
POST /orders/save       -> { orderId }
POST /order/pay         -> { success }
GET  /order/{id}        -> { orderId, status: "PAID" }
```

## 八、MVP 流程图

```
阶段一（闭环骨架）
├── 用户登录 ──→ 商品查询 ──→ 创建订单 ──→ 订单支付 ──→ 查询订单状态
└── 验证：单条流程可跑通

阶段二（稳定交易）
├── 加入注册、库存校验、订单取消、订单明细、全局异常
└── 验证：主流程稳定，异常处理完善

阶段三（统一入口）
├── 网关 + Sa-Token 认证 + 路由 + 限流/日志/跨域
└── 验证：统一入口访问，未登录拦截

阶段四（补齐 P0）
├── DTO/VO 规范化 + 日志 + RBAC 权限 + 回归测试
└── 验证：P0 全部完成
```

## 九、风险管理

| 风险 | 阶段 | 影响 | 应对 |
|-----|------|------|------|
| 阶段一任务量多，闭环时间拉长 | 阶段一 | 中 | 阶段一内部按天拆分：第1天数据库+用户服务，第2天商品+订单，第3天支付+联调 |
| 阶段一支付闭环失败 | 阶段一 | 高 | 先做 mock 支付，确认流程后再对接真实余额扣减 |
| Feign 调用超时或失败 | 阶段一 | 高 | 配置合理超时和重试，先做本地联调 |
| 数据库表结构变更 | 阶段一 | 中 | 预留扩展字段，避免后续大改 |
| 网关认证阻塞阶段一/二演示 | 阶段三 | 低 | 阶段一、二直连服务开发，阶段三再接入网关 |
