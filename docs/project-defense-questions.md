## 2. 功能问题

### 问题6
**问题**：登录流程是怎样的？

**提问意图**：验证你是否理解认证链路。

**标准回答**：
用户提交用户名和密码后，后端校验账号密码，成功后生成 JWT Token 返回给前端。前端后续请求携带 Token，Gateway 和后端服务共同完成身份校验。

**加分回答**：
登录态可以配合 Redis 做缓存或黑名单控制，便于失效和安全管理。

**扣分回答**：
登录就是点按钮，后端自动通过。

---

### 问题7
**问题**：订单流程怎么实现？

**提问意图**：验证交易主流程是否清楚。

**标准回答**：
用户选择商品后创建订单，系统校验库存并生成订单主表和订单明细表，随后进入支付流程，支付成功后更新订单状态。

**加分回答**：
创建订单时还会考虑幂等、防重复提交和库存一致性问题。

**扣分回答**：
下单就是插一条记录。

---

### 问题8
**问题**：AI 模块怎么实现？

**提问意图**：检查你是否理解 AI 与业务结合方式。

**标准回答**：
识别模块使用 YOLOv11 处理图片，推荐模块用 DIN 结合用户行为序列输出 Top-N，问答模块用 Neo4j 知识图谱完成实体识别和 Cypher 查询生成。

**加分回答**：
三个 AI 模块分别对应“看图找货”“兴趣推荐”“语义问答”，覆盖了购物场景中的三种典型交互。

**扣分回答**：
AI 部分就是调接口返回结果。

上传链路还要考虑文件类型校验、大小限制、异常处理和超时控制。

**扣分回答**：

## 3. 技术问题

### 问题11
**问题**：为什么用 Spring Boot 和 Spring Cloud Alibaba？

**提问意图**：验证技术选型是否有理由。

**标准回答**：
因为项目需要微服务治理能力，Spring Boot 负责快速构建服务，Spring Cloud Alibaba 提供 Nacos、Gateway、Sentinel 等组件，适合做完整的企业级架构。

**加分回答**：
它的优势是生态完整、配置统一、易于服务拆分，也方便和 Java 业务代码协同开发。

**扣分回答**：
因为这个框架比较流行。

---

### 问题12
**问题**：为什么用 Redis？

**提问意图**：验证缓存和高并发设计意识。

**标准回答**：
Redis 用来缓存热点商品、用户登录态、推荐结果和幂等 Token，可以减轻数据库压力并提升响应速度。

**加分回答**：
Redis 还可以支持分布式锁和限流辅助，是系统稳定性的重要基础。

**扣分回答**：
因为 Redis 很快，所以就用了。

---

### 问题13
**问题**：为什么用 JWT？

**提问意图**：验证认证方案是否合理。

**标准回答**：
JWT 适合前后端分离场景，不需要服务端长期保存会话状态，便于分布式部署和跨服务认证。

**加分回答**：
JWT 还能减少 Session 集中存储带来的扩展问题，但要配合过期控制和黑名单机制防止被盗用。

**扣分回答**：
JWT 比较高级，所以用了。

---

### 问题14
**问题**：为什么不用 Session？

**提问意图**：追问你是否理解 JWT 的替代方案。

**标准回答**：
Session 更适合单体或强状态服务，而当前项目是前后端分离和微服务架构，JWT 更适合分布式场景。

**加分回答**：
如果必须结合 Session，也可以把会话统一放到 Redis，但 JWT 更轻量，链路更简单。

**扣分回答**：
Session 过时了。

**提问意图**：验证 ORM 选型是否结合业务场景。

**标准回答**：
MyBatis Plus 更适合控制 SQL 细节，方便处理复杂查询、联表查询和性能优化，比较符合当前项目的业务特点。

**加分回答**：
对于电商类系统，SQL 可控性很重要，MyBatis Plus 在开发效率和可控性之间平衡得比较好。

**扣分回答**：
因为我更熟 MyBatis。

### 压力追问链 1：JWT 相关

**Q**：为什么用 JWT？

**A**：因为它适合前后端分离和分布式部署，不需要服务端维护大量会话状态。

**Q**：为什么不用 Session？

**A**：Session 更依赖服务端状态，不太适合当前这种微服务和分离部署场景。

**Q**：JWT 失效怎么办？

**A**：通过过期时间控制，同时结合刷新机制或重新登录处理。

**Q**：JWT 被盗怎么办？

**A**：可以缩短有效期、做黑名单控制、结合 HTTPS 传输和设备绑定策略。

**Q**：为什么不用 OAuth2？

**A**：OAuth2 更适合第三方授权和复杂身份体系，本项目是内部业务认证，JWT 已足够满足需求。

---

### 压力追问链 2：Redis 相关

**Q**：为什么用 Redis？

**A**：因为它适合做热点数据缓存、登录态和幂等 Token 存储，能提升系统性能。

**Q**：缓存失效怎么办？

**A**：可以采用 Cache Aside 策略，先查缓存，未命中再查数据库并回填。

**Q**：缓存和数据库不一致怎么办？

**A**：更新数据库后及时失效缓存，必要时结合延迟双删或消息通知。

**Q**：Redis 挂了怎么办？

**A**：核心链路仍可回退到数据库，只是性能下降；同时可以通过集群和持久化降低风险。

---

### 压力追问链 3：AI 相关

**Q**：为什么要把 YOLO 独立成一个服务？

**A**：因为它依赖 Python 和深度学习环境，且计算密集，独立部署更容易维护和扩展。

**Q**：识别结果不准怎么办？

**A**：可以从数据增强、标注质量、阈值设置和类别映射规则四个方面优化。

**Q**：推荐为什么不用简单协同过滤？

**A**：因为协同过滤无法很好地结合当前候选商品和用户历史行为的细粒度关系，DIN 的注意力机制更适合这个场景。

**Q**：KBQA 为什么不用关键词搜索？

**A**：关键词搜索只能做表面匹配，知识图谱可以表达实体关系，语义更强，也更利于回答结构化问题。

---

### 压力追问链 4：工程与扩展

**Q**：如果订单高并发怎么办？

**A**：先做缓存、限流、索引和幂等控制，再根据压力情况做服务拆分和资源扩容。

**Q**：如果库存被超卖怎么办？

**A**：下单和扣库存要在业务上做一致性校验，可以使用乐观锁或分布式锁控制。

**Q**：如果服务调用失败怎么办？

**A**：通过超时控制、Fallback 降级、重试策略和异常捕获提升系统容错能力。

**Q**：如果系统要上线，你还会补什么？

**A**：会补充监控告警、日志聚合、灰度发布、自动化测试和更完整的容灾方案。
