# 商品推荐模块实施方案（DIN + 前端联动）

## 一、项目目标

基于 `d:\Programming\YQSX\DIN` 中的 DIN 推荐算法，以及 `d:\Programming\YQSX\XML\recommend` 中提供的两份数据集，搭建一套可演示、可联调、可扩展的商品推荐模块。

本方案采用“应用层 + 算法层”协作模式：
- 应用层负责登录注册、用户 ID 映射、算法接口调用、推荐结果展示。
- 算法层负责 DIN 模型训练、推理与推荐结果返回。
- 前端风格参考 `d:\Programming\YQSX\docs\前端参考`，保持整体主题色调一致。

## 二、整体架构与业务流程

### 1. 角色职责划分

- **应用层**：负责用户端交互、登录鉴权、用户 ID 映射、调用算法接口、前端结果展示，全程不接触推荐算法逻辑，只处理脱敏 ID 数据。
- **算法层**：负责接收用户 ID、执行推荐算法计算、返回 Top10 脱敏商品 ID，不接触前端页面与用户登录态。

### 2. 完整执行链路

1. 用户在应用层页面完成登录。
2. 应用层从指定脱敏数据集中匹配对应用户 ID。
3. 应用层将脱敏用户 ID 作为参数，请求算法层的推荐接口。
4. 算法层基于该用户 ID，通过推荐模型计算出 Top10 商品（假设 10 个，可以更多）。
5. 算法层仅向应用层返回 10 个纯数字形式的脱敏商品 ID。
6. 应用层前端页面直接展示这 10 个数字 ID，不做任何商品信息还原。

## 三、数据与实现逻辑

### 1. 数据来源

- 用户行为数据：`d:\Programming\YQSX\XML\recommend\tianchi_mobile_recommend_train_user.csv`
- 商品信息数据：`d:\Programming\YQSX\XML\recommend\tianchi_mobile_recommend_train_item(1).csv`
- 现有训练产物参考：
  - `din_model.pt`
  - `train_samples.pkl`
  - `test_samples.pkl`
  - `user_behaviors.pkl`
  - `item_stats.pkl`
  - `vocab.pkl`

### 2. DIN 训练与推理逻辑

- 以用户历史行为序列作为 query 历史输入。
- 以候选商品作为 target 输入。
- 利用 DIN 的注意力机制建模用户对不同历史行为的兴趣偏好。
- 输出用户对候选商品的点击 / 转化概率。
- 推理阶段按候选商品集合逐个打分，排序后取 Top10。

### 3. ID 体系约束

- 统一使用数据集原生 `user_id` 与 `item_id`。
- 不自行生成新 ID。
- 不进行二次映射或业务侧 ID 转换。
- 应用层与算法层共享同一套脱敏 ID 体系，避免错配。

## 四、应用层详细实现

### 1. 登录与用户 ID 映射模块

- 提供登录入口，支持账号密码 / 学号登录，登录成功后生成用户登录态（token / session）。
- 内置“登录账号 ↔ 数据集用户 ID”映射表：将系统登录账号与你拿到的脱敏数据集中的用户 ID 一一绑定，登录成功后自动从数据集中取出对应用户 ID，全程前端无感知。
- 严格使用给定数据集原生的用户 ID 字段值，不自行生成、不做二次转换，保证与算法层 ID 体系完全一致。

### 2. 算法层接口调用模块

- 登录成功并获取到脱敏用户 ID 后，由应用层后端向算法层发起 HTTP 请求。
- 请求仅携带核心参数 `userId`，不传递任何用户真实信息。
- 接收算法层返回的推荐结果，解析为商品 ID 数组后传递给前端。

### 3. 前端脱敏展示模块

- 页面设置“个性化推荐”展示区域，仅做纯文本列表渲染。
- 每条推荐内容只显示数字形式的商品 ID，不包含商品名称、图片、价格、品类等任何真实信息。
- 推荐结果页同时保留“热门商品”与“推荐商品”两个展示区域，推荐区展示算法返回内容，热门区展示静态或聚合后的脱敏数字 ID。

### 4. 前端页面规划

- 登录页
- 注册页
- 首页 / 仪表盘页
- 热门商品页
- 推荐商品页
- 结果空态 / 异常提示页

### 5. 视觉设计原则

- 参考 `docs/前端参考` 中的 Hero / Auth / Pixel perfect 风格，采用统一主色调。
- 页面结构偏简洁、卡片化、层次清晰。
- 推荐结果区域采用大字号数字列表或卡片列表，强调“脱敏 ID”展示。
- 保持登录注册页与推荐页的视觉一致性，避免风格割裂。

## 五、算法层详细实现

### 1. 接口接收模块

- 提供标准化 HTTP 推荐接口，接收应用层传入的脱敏 `userId`。
- 仅做参数格式校验，不存储、不追溯用户真实身份。

### 2. 推荐计算模块

- 基于同一份脱敏数据集，使用部署好的推荐算法（DIN）对传入的用户 ID 进行推荐计算。
- 计算过程仅使用数据集内的脱敏用户 ID、商品 ID 及行为特征，不引入外部真实商品信息。

### 3. 结果返回模块

- 计算完成后，截取 Top10 推荐结果，仅输出纯数字商品 ID 数组。
- 不返回任何商品属性、描述、分类信息，保持数据完全脱敏状态。

## 六、接口联调统一规范

### 1. 请求格式（应用层 → 算法层）

```json
{
  "userId": 123456
}
```

### 2. 返回格式（算法层 → 应用层）

```json
{
  "code": 200,
  "msg": "success",
  "data": {
    "recommendList": [1001, 1002, 1003, 1004, 1005, 1006, 1007, 1008, 1009, 1010]
  }
}
```

### 3. 接口约束

- 只传递 `userId`，不传递手机号、姓名、邮箱等真实信息。
- 返回结果仅限数字商品 ID。
- 若结果不足 10 个，按实际可得数量返回，同时补足前端兜底提示。

## 七、核心规则与注意事项

1. **ID 体系统一**：两层必须共用同一份数据集的用户 ID、商品 ID，禁止各自做 ID 转换，否则会出现 ID 不匹配问题。
2. **脱敏边界严格**：应用层全程只处理数字 ID，不做商品信息还原；算法层只输出商品 ID，不返回业务详情，模拟数据隐私保护的联邦推荐场景。
3. **异常兜底**：若算法层接口超时、调用失败或返回数量不足 10 个，应用层前端需做友好提示，避免页面空白或报错。
4. **业务说明**：演示时可明确说明——该流程中 ID 均为脱敏数据，仅原始数据持有方（如平台）知晓 ID 对应的真实用户与商品信息，两层交互全程不泄露隐私数据。

### 阶段 3：联调与验证
- 校验用户 ID 映射是否与算法层一致。

## 九、风险分析

- **数据字段不一致风险**：原始 CSV 文件字段命名或格式可能与脚本假设存在差异，需要先核对。
- **样本构造偏差风险**：行为标签映射若不准确，可能影响 DIN 训练效果。
- **接口契约风险**：应用层与算法层若未统一 `userId` / `recommendList` 结构，会造成联调失败。
- **展示一致性风险**：前端若未统一主题色与组件风格，整体页面会显得割裂。

## 十、验证方案

- 校验异常场景下是否具备兜底提示。

## 十二、开发任务清单

### 1. 算法层开发任务清单

1. **数据核对与清洗**
   - 核对 `user.csv` 与 `item.csv` 字段结构。
   - 确认 `user_id`、`item_id`、行为类型、时间字段的含义。
   - 清理异常值、空值、重复行为记录。

2. **样本构造**
   - 按用户行为时间排序。
   - 构造 DIN 所需的历史序列样本、候选商品样本与标签。
   - 处理正负样本比例问题，必要时补充负采样。

3. **特征编码**
   - 统一用户 ID、商品 ID 的词表映射。
   - 预留 padding ID 与未知 ID 处理策略。
   - 固化序列最大长度与特征维度。

4. **DIN 模型训练**
   - 完成 embedding 层、attention pooling、DNN 分类头搭建。
   - 配置损失函数、优化器、AUC / Precision / Recall 评估指标。
   - 训练并保存最优模型权重。

5. **推理服务封装**
   - 封装用户历史序列读取逻辑。
   - 封装候选商品打分逻辑。
   - 提供输入 `userId` 返回 Top10 商品 ID 的推理函数。

6. **接口服务实现**
   - 提供标准 HTTP API。
   - 完成参数校验、异常处理、结果排序与 Top10 截取。
   - 输出统一 JSON 响应结构。

7. **模型与数据产物管理**
   - 固化训练产物目录。
   - 保存词表、样本集、模型文件、配置文件。
   - 支持后续快速复现与增量训练。

8. **算法层验证**
   - 验证输入不同 `userId` 是否可返回稳定结果。
   - 验证接口超时、空历史、未知用户等边界情况。
   - 验证返回值始终满足纯数字商品 ID 约束。

### 2. 应用层页面开发任务清单

1. **登录页**
   - 搭建账号密码 / 学号登录表单。
   - 登录成功后写入 token / session。
   - 完成与用户 ID 映射的入口逻辑。

2. **注册页**
   - 搭建注册表单。
   - 提供基础校验与提交反馈。
   - 保持与登录页一致的视觉风格。

3. **首页 / 仪表盘页**
   - 展示系统概览、推荐入口、用户状态。
   - 放置热门商品与推荐商品的快捷入口。

4. **热门商品页**
   - 展示固定或聚合后的脱敏商品 ID 列表。
   - 保持与推荐页统一的卡片风格。

5. **推荐商品页**
   - 登录后自动获取映射后的脱敏用户 ID。
   - 调用算法层接口并展示 Top10 推荐商品 ID。

7. **前端主题统一**
   - 参考前端参考稿统一色调、排版、卡片、按钮样式。

8. **页面联调**

   - 统一推荐列表渲染格式。
   - 验证登录态、用户 ID、推荐结果展示链路完整。

### 3. 接口字段定义稿

#### 3.1 登录接口

**请求**

```json
{
  "account": "student_001",
  "password": "******"
}
```

**返回**

```json
{
  "code": 200,
  "msg": "success",
  "data": {
    "token": "token_xxx",
    "userId": 123456
  }
}
```

**字段说明**
- `account`：登录账号，可为学号或系统账号。
- `password`：登录密码。
- `token`：登录态凭证。
- `userId`：映射后的脱敏数据集用户 ID。

#### 3.2 注册接口

**请求**

```json
{
  "account": "student_001",
  "password": "******",
  "confirmPassword": "******"
}
```

**返回**

```json
{
  "code": 200,
  "msg": "success",
  "data": {
    "userId": 123456
  }
}
```

**字段说明**
- `confirmPassword`：二次确认密码。
- `userId`：注册成功后绑定的脱敏用户 ID。

#### 3.3 推荐接口

**请求**

```json
{
  "userId": 123456
}
```

**返回**

```json
{
  "code": 200,
  "msg": "success",
  "data": {
    "recommendList": [1001, 1002, 1003, 1004, 1005, 1006, 1007, 1008, 1009, 1010]
  }
}
```

**字段说明**
- `userId`：应用层传入的脱敏用户 ID。
- `recommendList`：算法层返回的脱敏商品 ID 数组，按推荐分数降序排列。

#### 3.4 热门商品接口

**请求**

```json
{
  "page": 1,
  "pageSize": 10
}
```

**返回**

```json
{
  "code": 200,
  "msg": "success",
  "data": {
    "itemList": [1001, 1008, 1015, 1020, 1033, 1042, 1055, 1078, 1099, 1101]
  }
}
```

**字段说明**
- `page`：分页页码。
- `pageSize`：每页数量。
- `itemList`：热门商品脱敏 ID 列表。

## 十三、结论

本方案采用“应用层负责交互与脱敏 ID 映射、算法层负责 DIN 推荐计算”的分层模式，既能满足推荐模块演示需求，也能保证接口边界清晰、职责分离明确、前后端协作可控。
