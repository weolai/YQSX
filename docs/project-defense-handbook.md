## 第一章 一句话讲清项目

### 一句话版本
本项目是一个面向零食电商场景的 AI 智能商城系统，
实现拍照识别、个性化推荐、知识图谱问答和电商交易闭环，
采用 Spring Cloud Alibaba 微服务架构和 Next.js 前后端分离架构。

### 三句话版本
1. 这个项目解决的是传统零食电商找货慢、推荐弱、交互不智能的问题。  
2. 我们把 YOLOv11 图像识别、DIN 个性化推荐和 KBQA 智能问答结合到购物流程里。  
3. 最终实现了从拍照识别到商品推荐，再到下单支付的完整闭环。

### 一分钟版本
这个项目围绕零食电商场景，做了一个 AI 智能商城。前端使用 Next.js 16 和 React 19，后端使用 Spring Cloud Alibaba 搭建微服务，AI 层分别实现了 YOLOv11 图像识别、DIN 推荐和 Neo4j KBQA 问答。系统不仅支持商品浏览、订单、支付这些基础电商功能，还能通过拍照识别商品、根据用户行为推荐商品，并通过知识图谱回答商品相关问题，所以它既有业务价值，也有技术深度。

## 第二章 项目总体架构

### 架构图（ASCII）
```
用户
  ↓
前端（Next.js / React）
  ↓
Gateway
  ↓
业务服务（商品 / 订单 / 用户 / 支付）
  ↓
AI服务（YOLO / DIN / KBQA）
  ↓
数据库（MySQL / Redis / Neo4j）
```

### 技术栈
- 前端：Next.js 16、React 19、TypeScript、Tailwind CSS、shadcn/ui
- 后端：Spring Boot、Spring Cloud Alibaba、Gateway、OpenFeign、Sentinel、Nacos、MyBatis Plus
- 数据库：MySQL 8.0+、Redis、Neo4j
- 部署：前后端分离，本地多服务启动，AI 服务独立部署

### 请求链路
1. 用户在前端发起请求。  
2. 请求进入 Gateway，完成路由、鉴权和限流。  
3. 请求转发到对应业务服务。  
4. 业务服务先查缓存，再查数据库。  
5. 识别、推荐、问答请求再调用 AI 服务。  
6. 结果统一返回前端展示。

### 模块关系
- 前端展示层：页面交互和数据展示
- 网关层：统一入口、认证、限流
- 业务服务层：商品、订单、用户、支付
- AI 服务层：YOLO、DIN、KBQA
- 基础设施层：Nacos、MySQL、Redis、监控组件

## 第四章 功能模块讲解

### 1. 用户模块
- **模块功能**：登录、注册、个人信息、地址管理。
- **业务流程**：输入账号密码 → 校验 → 签发 JWT → 进入系统。
- **涉及表**：t_user、t_user_address。
- **涉及接口**：登录、注册、获取用户信息。
- **涉及类**：UserController、UserService、UserMapper。
- **答辩讲法**：用户模块是整个系统的入口，核心是身份认证和个人信息管理。

### 2. 商品模块
- **模块功能**：商品列表、分类、详情、库存。
- **业务流程**：前端请求 → 缓存命中优先 → 数据库兜底。
- **涉及表**：t_product、t_category。
- **涉及接口**：商品列表、商品详情、库存扣减。
- **涉及类**：ProductController、ProductService、ProductMapper。
- **答辩讲法**：商品模块承担了整个商城的展示和交易基础。

### 3. 订单模块
- **模块功能**：创建订单、查询订单、取消订单、状态流转。
- **业务流程**：选商品 → 扣库存 → 建订单 → 支付 → 更新状态。
- **涉及表**：t_order、t_order_item。
- **涉及接口**：创建订单、订单列表、取消订单。
- **涉及类**：OrderController、OrderService、OrderMapper。
- **答辩讲法**：订单模块是交易链路的核心，重点是幂等和一致性。

### 4. 支付模块
- **模块功能**：支付创建、支付确认、回调处理。
- **业务流程**：创建支付单 → 支付回调 → 更新订单状态。
- **涉及表**：t_payment、t_order。
- **涉及接口**：支付创建、支付回调。
- **涉及类**：PaymentController、PaymentService、PaymentMapper。
- **答辩讲法**：支付模块重点解决重复支付和状态同步问题。

### 5. YOLO 识别模块
- **模块功能**：上传图片识别零食类别。
- **业务流程**：上传图片 → 模型推理 → NMS 后处理 → 返回识别结果。
- **涉及表**：t_detection_log。
- **涉及接口**：POST /api/recognize。
- **涉及类**：FastAPI 接口、YOLO 模型、类别映射表。
- **答辩讲法**：这个模块让用户可以直接“拍照找商品”。

### 6. DIN 推荐模块
- **模块功能**：基于行为序列推荐商品。
- **业务流程**：收集行为 → 召回 → 精排 → 输出 Top-N。
- **涉及表**：t_user_behavior、t_product。
- **涉及接口**：推荐接口、缓存接口。
- **涉及类**：DIN 模型、推荐服务、缓存模块。
- **答辩讲法**：这个模块解决“猜你喜欢什么”的问题。

### 7. KBQA 模块
- **模块功能**：知识图谱问答。
- **业务流程**：问题输入 → 实体识别 → 意图识别 → Cypher 生成 → Neo4j 查询。
- **涉及表**：Neo4j 图谱数据。
- **涉及接口**：问答接口。
- **涉及类**：KBQA 服务、实体识别模块、图查询模块。
- **答辩讲法**：这个模块让系统具备自然语言交互能力。

## 第六章 核心技术展开

### Spring Boot / Spring Cloud Alibaba
- **为什么使用**：适合企业级微服务，生态完整。
- **如何实现**：Spring Boot 承载业务服务，Nacos 做注册配置，Gateway 做统一入口，Sentinel 做限流。
- **项目中的位置**：后端核心服务层。
- **答辩怎么讲**：这是系统稳定性和扩展性的基础。
- **老师会问什么**：为什么不用单体？为什么选阿里微服务组件？

### Redis
- **为什么使用**：缓存热点数据、登录态、幂等 Token。
- **如何实现**：Cache Aside、Token 过期控制、推荐结果预计算缓存。
- **项目中的位置**：缓存层。
- **答辩怎么讲**：Redis 让系统更快，也能降低数据库压力。
- **老师会问什么**：缓存和数据库不一致怎么办？

### JWT
- **为什么使用**：适合前后端分离和分布式认证。
- **如何实现**：登录签发 Token，后续请求携带 Bearer Token。
- **项目中的位置**：认证层。
- **答辩怎么讲**：JWT 让认证无状态，便于微服务共享身份。
- **老师会问什么**：为什么不用 Session？Token 被盗怎么办？

### MyBatis Plus
- **为什么使用**：SQL 可控，适合复杂查询和性能优化。
- **如何实现**：Mapper + Service 分层，复杂查询可手写 SQL。
- **项目中的位置**：数据访问层。
- **答辩怎么讲**：比纯 ORM 更能控制电商系统的复杂查询。
- **老师会问什么**：为什么不用 JPA？

### AI 模块
- **为什么使用**：项目核心亮点，增强业务价值。
- **如何实现**：YOLO 识别、DIN 推荐、KBQA 问答三条链路独立实现。
- **项目中的位置**：AI 服务层。
- **答辩怎么讲**：AI 不是装饰，而是嵌入了购物主链路。
- **老师会问什么**：为什么这样拆？怎么保证性能？

## 第十章 翻车预案

### 问题：系统打不开
**原因**：服务未启动、端口冲突、环境变量缺失。

**处理方式**：优先启动网关、数据库、Redis、Nacos，再启动业务服务。

**现场话术**：
“我这边先切换到已经准备好的架构图和录制结果，当前系统启动链路我已经提前验证过。”

### 问题：数据库异常
**原因**：连接配置错误、表未初始化、账号权限不足。

**处理方式**：检查配置文件、初始化 SQL、数据库账号权限。

**现场话术**：
“这个问题属于环境问题，不影响我对数据库设计和表结构的说明。”

### 问题：Redis 异常
**原因**：Redis 未启动、端口占用、缓存配置错误。

**处理方式**：临时关闭缓存链路，优先保证主流程可演示。

**现场话术**：
“Redis 主要用于性能优化，当前先用数据库链路演示核心业务。”

### 问题：权限异常
**原因**：Token 过期、角色不匹配、接口鉴权失败。

**处理方式**：重新登录或切换到公开接口展示。

**现场话术**：
“这里是鉴权链路在工作，说明系统权限控制是生效的。”

### 问题：AI 不可用
**原因**：模型未加载、Python 服务未启动、依赖缺失。

## 第十五章 15分钟讲稿

老师们好，我汇报的项目是 YQSX 智能零食商城。这个项目是围绕零食电商场景做的一个 AI 智能商城系统，核心目标是解决传统电商找货慢、推荐弱、交互不智能的问题。

项目目标主要有四个：第一，完成商品拍照识别；第二，完成个性化推荐；第三，完成知识图谱问答；第四，完成商品、订单、支付和用户的一整套电商功能。为了支撑这些能力，前端使用 Next.js 16 和 React 19，后端使用 Spring Cloud Alibaba 搭建微服务，AI 层则用 Python 生态来实现识别、推荐和问答。

从架构上看，系统采用前后端分离加微服务的混合架构。用户先在前端发起请求，请求经过 Gateway 统一入口后，进入商品、订单、用户和支付等业务服务。如果请求涉及识别、推荐或问答，就再调用独立的 AI 服务。这样设计的原因是各模块职责差异很大，拆开以后更利于独立部署、独立优化和后续扩展。基础设施层则包括 Nacos、MySQL、Redis、Sentinel、Zipkin 和 Prometheus 等组件，用来支撑注册配置、缓存、限流和监控。

数据库方面，我按照微服务数据库隔离原则设计了核心表。商品服务有商品表和分类表，用户服务有用户表和地址表，订单服务有订单表和订单明细表，支付服务有支付记录表，AI 模块还记录了用户行为表和识别日志表。这里我特别注意了唯一索引、联合索引和审计字段的设计，比如用户名、手机号、订单号要唯一，user_id 和 status 这类组合查询要建立联合索引，这样既能保证数据规范，也能提高查询效率。

核心功能模块可以分成七块：用户、商品、订单、支付、YOLO 识别、DIN 推荐和 KBQA 问答。用户模块负责登录注册和地址管理，认证使用 JWT；商品模块负责列表、分类、详情和库存；订单模块负责创建订单、查询和取消，并通过幂等 Token、防重复提交和乐观锁来保证一致性；支付模块负责支付创建和回调更新；YOLO 模块把用户上传的图片识别成商品类别；DIN 模块根据用户历史行为生成推荐结果；KBQA 模块通过 Neo4j 把自然语言问题转换成图查询，实现智能问答。

在技术实现上，我重点做了几个事情。第一，权限体系上采用 JWT 和 RBAC，由 Gateway 做统一鉴权，避免越权访问；第二，缓存体系上用 Redis 缓存热点商品、登录态和推荐结果，同时也用来存幂等 Token；第三，文件上传链路中先校验图片格式和大小，再送给 YOLO 服务推理；第四，AI 服务与业务服务独立部署，避免深度学习环境影响主业务；第五，数据访问层使用 MyBatis Plus 来控制 SQL 细节，方便处理复杂查询和性能优化。

这个项目的亮点我总结有五个。第一，AI 和电商业务结合得比较紧，不是单独做模型展示，而是直接接到了购物流程里；第二，YOLO、DIN、KBQA 三类 AI 能力覆盖了“看图找货”“猜你喜欢”“语义问答”三个典型场景；第三，微服务架构完整，包含网关、注册中心、限流和监控；第四，数据库设计比较规范，考虑了索引和隔离；第五，系统里加入了限流、幂等、缓存和监控等工程化能力，更接近真实项目。

项目的难点主要有三类。第一，AI 识别结果怎么准确映射到商品，这里我做了类别映射和商品关联规则；第二，多服务协同怎么保证一致性，这里用了幂等 Token、乐观锁和分布式锁思路；第三，推荐系统怎么做到既准又快，这里用了召回加精排的两阶段方案，再结合预计算缓存提升性能。

安全方面，系统做了 JWT 认证、RBAC 授权、SQL 注入防护、XSS/CSRF 防护和数据隔离。性能方面，主要做了缓存优化、索引优化、分页优化和推荐预计算。测试方面，我也分别做了单元测试、接口测试、AI 模型测试和性能测试，确保系统能够稳定运行。
