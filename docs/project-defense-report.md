## 第二章 项目概述

### 项目背景

传统零食电商存在三个典型问题：
1. 用户找货成本高，图片与商品之间缺少快速映射。
2. 推荐能力弱，无法基于用户兴趣精准推荐。
3. 智能问答缺位，用户在选购过程中缺少自然语言辅助。

因此，本项目引入 YOLOv11 图像识别、DIN 个性化推荐和 KBQA 智能问答能力，形成“拍照识别 + 智能推荐 + 在线购买”的完整闭环。

### 项目目标

- 实现零食商品的拍照识别与自动分类。
- 实现基于用户行为的个性化推荐。
- 实现基于知识图谱的商品问答。
- 实现商品、订单、支付、用户等核心电商功能。
- 实现可用于答辩展示的完整工程闭环。

### 用户角色

普通用户：
- 浏览商品、拍照识别、加入购物车、下单支付、查看订单。

管理员：
- 管理商品、分类、库存、订单、支付记录与系统配置。

其他角色：
- 平台运营人员：维护推荐素材、商品展示和活动信息。
- 答辩老师：关注架构、实现、价值与个人贡献。

### 一句话总结

本项目是一个面向零食电商场景的 AI 智能商城系统，采用 Spring Cloud Alibaba 微服务架构与 Next.js 前端架构，实现 YOLO 识别、DIN 推荐、KBQA 问答和完整电商交易能力。

## 第三章 系统总体架构

### 技术栈

前端：Next.js 16、React 19、TypeScript、Tailwind CSS、shadcn/ui

后端：Spring Boot、Spring Cloud Alibaba、Gateway、OpenFeign、Sentinel、Nacos、MyBatis Plus

数据库：MySQL 8.0+、Redis、Neo4j

缓存：Redis、Caffeine（本地热点缓存）

部署：前后端分离，本地多服务启动，AI 服务独立部署

### 系统架构图

```
用户
 ↓
前端（Next.js 16 / React 19）
 ↓
Controller / API Route
 ↓
Service
 ↓
Mapper / Repository
 ↓
数据库（MySQL / Redis / Neo4j）
```

### 请求链路

1. 用户在前端页面发起操作。
2. 请求进入 Gateway，完成路由、鉴权和限流。
3. 请求转发至对应业务服务。
4. 业务服务先查缓存，再查数据库。
5. 涉及识别、推荐、问答时调用 AI 服务。
6. 业务结果统一返回前端展示。

```

  ↓

  ↓
业务服务（商品 / 订单 / 用户 / 支付）
  ↓
AI服务（YOLO / DIN / KBQA）
  ↓
数据库（MySQL / Redis / Neo4j）
```

## 第五章 核心功能模块

### 5.1 用户模块

功能目标：登录、注册、个人信息、地址管理。

业务流程：用户提交登录信息 → 校验账号密码 → 生成 JWT → 返回前端。

核心接口：登录、注册、获取个人信息、地址增删改查。

涉及表：t_user、t_user_address。

涉及类：UserController、UserService、UserMapper。

实现逻辑：JWT 认证 + RBAC 授权 + Redis Session 辅助。

### 5.2 商品模块

功能目标：商品展示、分类筛选、库存管理、商品详情。

业务流程：前端请求商品列表 → 服务查询缓存/数据库 → 返回商品数据。

核心接口：商品列表、商品详情、商品新增、商品更新、库存扣减。

涉及表：t_product、t_category。

涉及类：ProductController、ProductService、ProductMapper。

实现逻辑：支持分类查询、热销排序、库存扣减和识别结果映射。

### 5.3 订单模块

功能目标：创建订单、查询订单、取消订单、订单状态流转。

业务流程：选择商品 → 生成订单 → 扣减库存 → 创建支付记录 → 返回订单号。

核心接口：创建订单、订单列表、订单详情、取消订单。

涉及表：t_order、t_order_item。

涉及类：OrderController、OrderService、OrderMapper。

实现逻辑：幂等 Token + 乐观锁 + 分布式锁防重。

### 5.4 支付模块

功能目标：发起支付、确认支付、回调处理、支付状态更新。

业务流程：提交支付请求 → 生成支付单 → 更新订单状态 → 回调确认。

核心接口：支付创建、支付查询、支付回调。

涉及表：t_payment、t_order。

涉及类：PaymentController、PaymentService、PaymentMapper。

实现逻辑：支付幂等、防重复回调、状态一致性控制。

### 5.5 YOLO识别模块

功能目标：识别上传的零食图片并返回候选商品。

业务流程：上传图片 → YOLOv11 推理 → NMS 后处理 → 商品匹配 → 返回结果。

核心接口：POST /api/recognize。

涉及表：t_detection_log。

涉及类：FastAPI recognize 接口、YOLO 推理模型、类别映射表。

实现逻辑：图片预处理 + 模型推理 + 识别结果映射。

### 5.6 DIN推荐模块

功能目标：根据用户历史行为生成个性化推荐。

业务流程：收集用户行为 → 召回候选集 → DIN 精排 → 输出 Top-N。

核心接口：推荐列表接口、用户推荐缓存接口。

涉及表：t_user_behavior、t_product。

涉及类：DIN 模型、推荐服务、缓存管理模块。

实现逻辑：召回 + 精排两阶段架构，提升性能与准确度。

### 5.7 KBQA模块

功能目标：支持自然语言商品问答。

业务流程：问题输入 → 实体识别 → 意图识别 → Cypher 生成 → Neo4j 查询 → 答案输出。

核心接口：问答接口、知识图谱查询接口。

涉及表：Neo4j 图谱节点与关系。

涉及类：KBQA 服务、实体识别模块、图查询生成模块。

实现逻辑：基于图谱的语义理解与结构化查询。

## 第六章 核心技术实现

### 权限体系

- JWT：前后端统一身份凭证。
- RBAC：普通用户、管理员、运营角色分权。

- 缓存商品详情和热点数据。

- 用户上传商品图片用于 YOLO 识别。
- 前端通过表单或接口上传至识别服务。
- 后端进行格式校验、大小限制和安全检查。

### 消息队列

- 项目架构预留异步解耦能力。
- 可用于订单创建后通知、库存变更广播、日志同步。

### AI模块

- YOLOv11：负责图像目标检测。
- DIN：负责个性化推荐。
- KBQA：负责知识图谱问答。

## 第七章 源码级讲解

### 文件：docs/paper-kit/06_yolo_design.md

作用：说明 YOLO 图像识别模块的训练、推理与接口实现。

核心方法：`/api/recognize` 识别接口、模型加载、后处理流程。

讲解重点：图片预处理、推理速度、mAP 指标、类别映射。

老师可能追问：为什么选择 YOLOv11？如何保证识别准确率？

标准回答：YOLOv11 在精度和速度之间较平衡，适合实时商品识别场景；通过数据增强、标注规范和置信度阈值控制提升效果。

### 文件：docs/paper-kit/07_din_design.md

作用：说明推荐系统的召回、精排与缓存策略。

核心方法：召回候选生成、DIN 评分、Top-K 输出、离线预计算。

讲解重点：注意力机制、召回+精排、缓存命中率。

老师可能追问：为什么不用简单协同过滤？

标准回答：协同过滤难以刻画当前候选商品与用户历史行为的相关性，DIN 用注意力机制动态建模，推荐精度更高。

### 文件：docs/paper-kit/08_kbqa_design.md

作用：说明知识图谱问答的实体识别、图查询和答案生成。

核心方法：问题解析、实体链接、Cypher 生成、Neo4j 查询。

讲解重点：图谱建模、查询模板、答案可解释性。

老师可能追问：为什么使用 Neo4j？

标准回答：Neo4j 更适合表达商品、分类、口味、推荐关系等复杂图结构，查询路径直观、扩展方便。

## 第九章 项目难点

### 难点1：AI识别结果与商品库映射

问题：识别模型输出的是类别标签，但业务需要返回具体商品。

解决方案：建立类别映射表与商品数据关联规则。

最终效果：识别结果可直接转化为商品推荐列表。

工程价值：增强识别能力的业务可用性。

### 难点2：多微服务协同与幂等控制

问题：订单、库存、支付跨服务调用容易出现重复或不一致。

解决方案：使用幂等 Token、乐观锁、分布式锁和状态机控制。

最终效果：减少重复下单与重复支付风险。

工程价值：提升系统稳定性与可靠性。

### 难点3：推荐系统性能与效果平衡

问题：全量精排耗时过长，无法在线使用。

解决方案：采用召回+精排两阶段方案，并引入缓存预计算。

最终效果：推荐响应时间从秒级优化到可在线服务。

工程价值：兼顾效果、性能与可扩展性。

## 第十章 性能优化

- 缓存优化：商品详情、推荐结果、登录态使用 Redis 缓存。
- SQL优化：避免 `SELECT *`，使用覆盖索引和联合索引。
- 分页优化：避免深度分页，使用 ID 范围分页。
- 索引优化：为高频查询字段建立唯一索引和联合索引。
- 并发优化：接口限流、异步化、推荐结果预计算。

## 第十四章 答辩总结

### 30秒版本

本项目是一个基于 Spring Cloud Alibaba 和 Next.js 的 AI 智能零食商城，核心创新点是将 YOLO 图像识别、DIN 个性化推荐和 KBQA 知识图谱问答整合到电商流程中，实现“拍照识别即购物”的完整体验。

### 1分钟版本

本项目面向零食电商场景，采用前后端分离和微服务架构，前端使用 Next.js 16，后端使用 Spring Cloud Alibaba，并集成 YOLOv11、DIN 和 Neo4j KBQA 三类 AI 能力。系统不仅支持商品、订单、支付等基础电商功能，还实现了拍照识别、个性化推荐和智能问答，兼顾业务价值、技术深度和工程完整性。

### 3分钟版本

本项目从真实电商痛点出发，解决用户找货难、推荐弱、交互不智能的问题。整体采用微服务架构，将前端、网关、业务服务、AI 服务和基础设施分层设计；数据库上采用独立数据库与索引优化；安全上采用 JWT、RBAC、限流、幂等等机制；AI 层则集成 YOLOv11、DIN 和 KBQA，分别解决图片识别、兴趣推荐和知识问答问题。项目最终实现了从识别、推荐、下单、支付到后台管理的完整业务闭环，也体现了我在架构设计、接口实现、数据库设计、AI 集成和工程落地方面的综合能力。
