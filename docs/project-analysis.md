# 项目分析报告

## 项目画像

项目名称：YQSX 智能零食商城

项目类型：企业级课程设计/毕业设计方向的 AI 电商系统

项目目标：
- 解决传统零食电商“找货难、推荐弱、识别慢”的问题
- 提供“拍照识别 + 智能推荐 + 在线下单”的闭环体验
- 形成可用于答辩展示的完整技术闭环：前端、后端、AI、数据库、测试、部署

用户群体：
- 终端消费者：拍照识别零食并快速购买
- 平台运营人员：维护商品、订单、库存、支付等业务
- 答辩评审老师：关注系统完整性、技术深度、工程落地能力

## 架构画像

架构模式：
- 前后端分离
- 微服务架构
- 分层架构
- AI 服务与业务服务解耦

模块划分：
- 前端展示层：商品浏览、拍照识别、订单管理、用户中心
- 网关层：统一入口、路由、认证、限流
- 业务服务层：商品、订单、用户、支付、购物车、库存
- AI 服务层：YOLO 识别、DIN 推荐、KBQA 问答
- 基础设施层：Nacos、MySQL、Redis、Zipkin、Prometheus、Sentinel

请求链路：
1. 用户在前端发起请求
2. 请求进入 Gateway 统一处理
3. Gateway 根据路由转发到对应微服务
4. 业务服务访问本地数据库或缓存
5. 涉及识别/推荐/问答时调用 AI 服务
6. 最终结果返回前端展示

## 数据画像

核心表：
- t_user：用户表
- t_product：商品表
- t_product_category：商品分类表
- t_order：订单表
- t_order_item：订单明细表
- t_cart：购物车表
- t_payment：支付记录表
- t_user_behavior：用户行为表
- t_detection_log：识别日志表

ER关系：
- 用户 1:N 地址
- 用户 1:N 订单
- 订单 1:N 订单明细
- 商品 1:N 订单明细
- 分类 1:N 商品
- 用户 1:N 购物车
- 商品 1:N 购物车

数据流：
- 前端请求进入业务服务后，先查询缓存，再访问 MySQL
- 识别服务将图片识别结果写入识别日志或返回候选商品
- 推荐服务根据用户行为数据生成候选集并排序
- KBQA 服务基于 Neo4j 图谱进行实体匹配与查询生成

---

## 业务画像

核心模块：
1. 用户模块：登录、注册、个人信息、地址管理
2. 商品模块：商品展示、分类、库存、详情
3. 订单模块：下单、取消、查询、状态流转
4. 支付模块：支付创建、支付确认、支付状态回调
5. AI 识别模块：拍照识别零食类别并返回推荐结果
6. 推荐模块：基于 DIN 的个性化推荐
7. KBQA 模块：基于知识图谱的智能问答

核心流程：
- 识别流程：上传图片 → YOLOv11 检测 → 商品匹配 → 返回商品列表
- 推荐流程：用户行为输入 → 候选召回 → DIN 排序 → 输出 Top-N
- 问答流程：自然语言提问 → 实体识别 → 图查询生成 → Neo4j 检索 → 生成答案
- 交易流程：浏览商品 → 加入购物车 → 创建订单 → 支付 → 完成交易

## 权限画像

认证：
- 统一登录入口
- JWT Token 作为身份凭证
- Gateway 统一验证 Token

授权：
- RBAC 权限模型
- 区分普通用户、管理员、运营角色
- 接口级权限控制

数据隔离：
- 每个微服务独立数据库
- 避免跨服务直接访问数据库
- 通过接口和服务调用完成数据协作

## 项目亮点

亮点1：AI 与电商业务深度结合，形成“拍照识别即购物”的完整闭环。

亮点2：同时集成 YOLOv11、DIN、KBQA 三类 AI 能力，技术覆盖面广，答辩展示价值高。

亮点3：采用 Spring Cloud Alibaba 微服务架构，具备网关、注册中心、限流、监控等完整工程能力。

---

## 项目难点

难点1：AI 服务与业务服务的解耦与联调，尤其是识别结果与商品库的映射关系。

难点2：多微服务之间的调用链路控制，包括认证、限流、异常处理与幂等设计。

难点3：推荐与问答模块需要同时满足技术深度与业务可解释性，输出结果要可展示、可讲解。

---

## 答辩价值总结

老师最可能关注：
1. 为什么选择微服务而不是单体架构
2. YOLOv11、DIN、KBQA 三个模块分别解决什么业务问题
3. 系统如何保证稳定性、安全性和可扩展性

个人贡献重点：
1. 完整系统架构设计与模块划分
2. AI 识别、推荐、问答能力集成
3. 数据库设计、接口设计、测试与部署方案
